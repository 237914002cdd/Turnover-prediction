## 2. 数据安全与合规（第二道防线的核心）

### 2.1 数据脱敏架构

Turnover ONA 系统从设计之初就遵循 **Privacy by Design** 原则。我们的核心承诺是：

> **绝不触碰员工聊天正文（Content）——仅处理元数据（Metadata）。**

```
┌─────────────────────────────────────────────────────────────┐
│                      数据源（客户侧）                           │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                   │
│  │ 飞书/企微 │  │  HRIS    │  │ 考勤系统  │    ...             │
│  └─────┬────┘  └────┬─────┘  └────┬─────┘                   │
│        │            │             │                          │
│        ▼            ▼             ▼                          │
│  ┌─────────────────────────────────────┐                     │
│  │      SHA-256 局部哈希脱敏层           │ ← 在客户侧执行      │
│  │  - employee_name → SHA-256(name)    │                     │
│  │  - 手机号/邮箱 → SHA-256(id)        │                     │
│  │  - Content 正文 → ❌ 不采集          │                     │
│  └──────────────┬──────────────────────┘                     │
│                 │                                            │
│                 ▼ 仅脱敏后的元数据                            │
│  ┌─────────────────────────────────────┐                     │
│  │     Turnover ONA Platform            │                     │
│  │  - ONA 图谱（匿名节点 + 边关系）       │                     │
│  │  - SHAP 归因（聚合特征贡献度）         │                     │
│  │  - ROI 沙盘（不涉及个人隐私）          │                     │
│  └─────────────────────────────────────┘                     │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 数据资产评估（数据冷启动清单）

实施团队需与客户 IT/HR 协作确认以下数据源的可用性：

| 数据类别 | 必要/可选 | 包含字段 | 脱敏要求 |
|---------|----------|---------|---------|
| **员工主数据** | 必要 | 工号、部门、职级、司龄 | 工号 MD5 → SHA-256 升级 |
| **薪酬数据** | 必要 | 月薪、薪资增幅、奖金 | 仅保留数值，不存银行账号 |
| **考勤数据** | 必要 | 月工作时长、加班标记、请假天数 | 不采集考勤明细位置 |
| **绩效数据** | 必要 | 绩效评分、晋升次数、培训时长 | 不采集 360 评语原文 |
| **协作日志** | 建议 | 跨部门消息频次、文档协作者 | ⚠️ 仅频次，不采集内容正文 |
| **招聘数据** | 可选 | 入职来源、面试反馈 | 不采集面试录音/录像 |
| **离职数据** | 建议 | 离职原因、离职日期 | 脱敏至原因分类级别 |

### 2.3 法务合规白皮书条款

> **以下条款需写入客户合同的"数据安全附件"中，作为具有法律约束力的承诺：**

1. **数据最小化原则**: Turnover ONA 仅采集实现离职风险预测所必需的最小字段集（39 字段宽表）。任何超出该范围的字段采集需经客户 IT 和法务双签审批。
2. **Content 绝对隔离**: 系统在物理层和逻辑层均不采集、不传输、不存储任何员工的即时消息正文、邮件正文、文档正文或语音内容。仅采集通信频次、协作关系等元数据。
3. **脱敏前置**: 所有个人标识（姓名、手机号、邮箱、工号）在进入系统之前完成 SHA-256 局部哈希脱敏，密钥由客户保管。我们仅存储哈希值，不具备反向还原能力。
4. **数据驻地**: SaaS 版本数据存储于指定云服务商（阿里云/腾讯云）的客户所选 Region。私有部署版本数据完全位于客户自管基础设施内。
5. **删除权**: 合同终止后 90 天内，客户可要求全量数据删除，我方提供书面删除确认函。
6. **审计权**: 合规级客户可要求第三方安全审计机构对系统进行渗透测试和代码审计，年审限一次。

## 3. 企业冷启动数据导入方案

### 3.1 通用场景：CSV/Excel 手动导入（Lite / Professional）

适用于缺乏自动化数据管道的中小型企业。

**步骤**:
```
Step 1: 客户 HR 从 HRIS/Excel 导出员工数据（39 字段模板）
        ↓
Step 2: 运行本地脱敏脚本（< 5 行 Python），对姓名/工号列执行 SHA-256
        ↓
Step 3: 在 ONA 平台点击"批量导入员工数据"
        ↓
Step 4: 系统返回插入结果：总行数、成功行数、跳过行数、错误列表
        ↓
Step 5: HR 确认数据无误 → ONA 图谱自动生成
```

**所需时间**: 从数据准备到图谱生成，单人操作 ≤ 30 分钟。

### 3.2 进阶场景：API 对接飞书/钉钉/企业微信（Enterprise + 增值服务）

适用于有内部开发团队的大型企业。

**架构**:
```
客户 HRIS/IM  →  ONA ETL Adapter（客户侧部署）  →  SHA-256 脱敏  →  HTTPS  →  ONA API
     ↑                        ↑
  数据源                 该适配器是无状态的，
  定时同步                仅做格式转换 + 脱敏
```

**API 端点**: `POST /api/v1/ona/graph/upload`（接受 CSV/XLSX）

**要求**:
- 客户需提供只读 API 凭证（无需管理员权限）
- 数据同步频率：日级（T+1）即可满足预警需求
- 企业版支持定制化调度策略（每小时/每天/每周）

### 3.3 高合规场景：物理隔离私有部署（Compliance-Grade）

适用于金融、医疗、政务等受监管行业。

| 要求 | 标准方案 | 可选增强 |
|------|---------|---------|
| 服务器位置 | 客户数据中心 | 双活/异地容灾 |
| 网络隔离 | 内网部署，无外部访问 | 物理断网（Air-Gap） |
| 数据库加密 | AES-256 静态加密 | 客户自管密钥（BYOK） |
| 审计日志 | 标准操作日志 | 完整数据血缘追踪 |
| 国产化适配 | 可选飞腾/鲲鹏 + 达梦/人大金仓 | 全部国产化组件 |

## 4. 组织变革管理（Change Management）

### 4.1 利益相关者矩阵

| 角色 | 关注点 | 沟通策略 |
|------|--------|---------|
| **CEO** | 人力资本 ROI、组织稳定性 | 展示总期望替换损失（¥200,064）与 ROI 沙盘 |
| **CFO** | 预算合理性、财务可量化 | 前置 ROI 测算——"投入 ¥15,600 净省 ¥2,520" |
| **CHO / HRD** | 组织洞察、创新性 | 竞品分析对比 + 被动 ONA 的差异化定位 |
| **HRBP** | 日常使用便捷性、数据可信度 | 现场 POC——带真实数据 30 分钟出报告 |
| **IT 部门** | 安全合规、部署复杂度 | 技术白皮书 + 脱敏架构 + 渗透测试报告 |
| **员工代表** | 隐私保护、数据使用透明度 | 全员公告解释仅采集元数据、不监控个人聊天 |

### 4.2 常见反对意见与应对话术

| 反对意见 | 应对话术 |
|---------|---------|
| "这不就是在监控员工吗？" | "我们只分析团队层面的协作频率，**不读任何聊天内容**。就像电梯里的监控摄像头拍得到谁在进出，但听不到谁在说话。隐私边界在物理层面就保障了。" |
| "我们已经买了 Culture Amp，不想重复投资。" | "Culture Amp 测量的是'员工怎么说'（调查），我们测量的是'员工怎么做'（行为）。两者是互补的。我们的客户通常并行使用——Culture Amp 用于文化洞察，ONA 用于离职预测。" |
| "数据不在我们系统里，没有协作日志怎么办？" | "我们有模板化的 CSV 导入方案，从 HRIS 导出→脱敏→上传，30 分钟出图谱。不需要任何系统对接就能跑起来。" |
| "调薪预算被冻住了，有洞察也没预算执行。" | "ROI 沙盘的价值恰恰在此——它帮你算清楚'不调薪的隐性成本'。拿着净节约 ¥2,520 的数字去和 CFO 谈预算重新分配，比'这是个核心人才'有说服力得多。" |

## 6. 术语表（Bilingual）

| 英文术语 | 中文术语 | 说明 |
|---------|---------|------|
| Privacy by Design | 设计即隐私 | 从系统设计阶段将隐私保护嵌入架构的原则 |
| SHA-256 Partial Hash | SHA-256 局部哈希 | 对个人标识进行不可逆脱敏处理 |
| Metadata | 元数据 | 描述数据的数据（如协作频次，而非内容本身） |
| Cold Start | 冷启动 | 从零开始导入数据建立 ONA 模型的过程 |
| ETL (Extract-Transform-Load) | 数据抽取-转换-加载 | 对接数据源的标准化流程 |
| Air-Gap Deployment | 物理隔离部署 | 服务器完全不连接互联网的最高安全方案 |
| BYOK (Bring Your Own Key) | 自管密钥 | 客户持有数据加密密钥，服务商无法自行解密 |
| Change Management | 变革管理 | 组织引入新系统时的人员变更管理流程 |
| POC (Proof of Concept) | 概念验证 | 客户的短期试用期，验证产品与需求的适配度 |
| TAM / CSM | 技术客户经理 / 客户成功经理 | 企业级售后的两层服务支持体系 |

| Right to Deletion | 删除权 | 合同终止后客户要求彻底删除数据的法定权利 |
| On-Premises | 本地部署 / 私有化 | 系统安装于客户自管服务器的部署模式 |
